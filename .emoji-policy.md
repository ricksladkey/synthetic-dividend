# Emoji Policy

## Why No Emojis in Documentation?

This project avoids emojis in documentation and code for technical reasons:

### Problems with Emojis

1. **Encoding Issues**
   - Windows terminals default to cp1252 encoding
   - Python print() fails with `UnicodeEncodeError`
   - Copy-paste from docs breaks in terminals

2. **Compatibility**
   - Not all terminals render emojis (SSH, old systems)
   - Different rendering across platforms
   - Breaks in some text editors

3. **Accessibility**
   - Screen readers struggle with emojis
   - Unclear meaning for vision-impaired users

4. **Professionalism**
   - Some enterprise environments frown on emojis
   - Technical documentation standards discourage them

## Guidelines

### DO

- Use clear, descriptive text
- Use markdown formatting (bold, italic, code blocks)
- Use ASCII art for visual elements
- Use `[OK]`, `[FAIL]`, `WARNING:`, `TIP:` prefixes

### DON'T

- Don't add emojis to .md files
- Don't use emojis in code comments
- Don't use emojis in CLI output
- Don't use emojis in commit messages (they break terminal display)

## Exception: Git Commit Messages

GitHub automatically strips emojis from commit messages that can't be displayed,
so the `ðŸ¤– Generated with Claude Code` footer gets cleaned up automatically.

For terminal display, we remove emojis manually before committing.

## Maintenance

If emojis accidentally get added, run:

```bash
python scripts/remove_emojis.py --dry-run   # Preview changes
python scripts/remove_emojis.py             # Apply changes
```
